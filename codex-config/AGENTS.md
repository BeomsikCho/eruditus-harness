# AGENTS.md

Codex 행동 가이드라인.
속도보다 정확성과 안전성을 우선한다.

---

## 1. 구현 전 사고 (Think Before Coding)

- 가정이 있다면 명확히 드러낸다.
- 해석이 여러 개 가능하면 간단히 나열한다.
- 더 단순한 방법이 있다면 제안한다.
- 이해되지 않는 부분이 있으면 구현 전에 질문한다.

---

## 2. 단순성 우선 (Simplicity First)

- 요청된 기능만 구현한다.
- 불필요한 abstraction을 만들지 않는다.
- 재사용되지 않는 코드를 일반화하지 않는다.
- 필요 없는 설정(config)이나 유연성을 추가하지 않는다.
- 불필요한 예외 처리를 하지 않는다.

기준:
> 과도하게 복잡해 보이면 단순화한다.

---

## 3. 최소 변경 원칙 (Surgical Changes)

- 필요한 부분만 수정한다.
- 관련 없는 코드, 주석, 포맷을 변경하지 않는다.
- 기존 스타일을 유지한다.

추가 규칙:
- 본인의 수정으로 인해 생긴 unused 코드만 제거한다.
- 기존에 있던 dead code는 제거하지 않는다.

기준:
> 모든 변경 라인은 사용자 요청과 직접적으로 연결되어야 한다.

---

## 4. 목표 기반 수행 (Goal-Driven Execution)

모호한 요청을 검증 가능한 목표로 변환한다.

예:
- "버그 수정" → 재현 → 수정 → 테스트 통과
- "검증 추가" → 실패 케이스 작성 → 통과

멀티 스텝 작업은 계획을 명시한다:

1. 작업 → 검증 방법
2. 작업 → 검증 방법

---

## 5. ADAS / C++ 안전성

대부분의 코드들은 safety-critical합니다.

- state machine, timing, threshold는 민감한 영역으로 취급한다.
- 명확한 지시 없이 동작 변경을 하지 않는다.
- implicit behavior보다 explicit logic을 선호한다.
- runtime 경로에서 숨겨진 allocation을 피한다.

안전 관련 코드가 불확실한 경우:
- 수정 전에 확인한다.

---

## 6. 검증 (Validation)

변경 후:

- 가능한 최소 범위로 build/test/lint 수행
- 실패 시 무시하지 않고 원인 분석
- 실행 불가능한 경우, 실행 방법 명시

---

## 7. Memory / RAG 사용

작업 중 이후에도 재사용할 가치가 있는 know-how를 발견하면 `.codex/memory`에 저장한다.

저장 대상:
- 프로젝트 고유 빌드/테스트 절차
- 반복적으로 필요한 디버깅 방법
- 문서에는 없지만 코드나 실행으로 확인한 제약
- 안전상 중요한 판단 기준

저장하지 않을 대상:
- 일회성 추측
- 검증되지 않은 가정
- 민감 정보, credential, token

수정 권한이 없는 회사 문서(`*.xml`, `*.pptx`, `*.docx`, `*.html`, `*.md`, `*.markdown`)는 원본을 변경하지 않고 RAG index를 통해 참조한다.

참조 문서 검색 시:
- 검색 결과의 source path와 chunk index를 확인한다.
- 검색 결과만으로 확정하지 말고 필요한 경우 원문 파일을 다시 확인한다.
- 참조 문서의 내용을 수정하지 않는다.

---

## 성공 기준

다음이 만족되면 올바르게 동작하는 것:

- 변경이 최소화됨
- 불필요한 복잡도가 없음
- 구현 전에 질문이 발생함
- 검증이 수행됨
